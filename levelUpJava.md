# Level Up Java（今日学习总结 - 面向 Java 后端初学者）

本文从“Java 后端初学者”的视角，总结今天在 **RL-Job-Scheduler** 项目里真正学到的东西：业务逻辑如何落地、架构为什么这么选、取舍的原因、实现细节、踩过的坑与解决方式，以及新接触到的技术栈。

---

## 1. 我真正理解了什么是“业务逻辑”

以前更像是“把功能写出来”，今天更像是在做一条完整的业务链路：**提交任务 -> 调度执行 -> 观测日志 -> 更新状态 -> 故障恢复**。

### 1.1 任务系统最核心的状态机
训练任务不只是一个函数调用，而是一条生命周期：
- `PENDING`：已入库/已入队，等待 Worker
- `RUNNING`：已经被某个 Worker 接管执行
- `COMPLETED / FAILED`：执行结束，产生最终结果或错误信息

今天最大的认知升级是：**状态机必须在“异常/宕机/重启”场景下仍然成立**。只处理“正常跑通”远远不够。

### 1.2 “被遗忘的任务”是怎么来的
我以前会觉得：调度失败就返回 `PENDING`，以后总会有人处理。
但实际系统不会“自动变好”：
- 如果没有“队列重试/对账”，那次失败就可能是最后一次尝试
- 任务看起来像“被系统忘记”

所以业务逻辑需要补上两类机制：
- **队列**：保证“以后会再试一次”
- **对账**：保证“即使队列丢了/异常了，也能从 DB 自愈”

## 2. 架构取舍：为什么我们最后选了这些方案

### 2.1 为什么要引入 Gateway（入口治理）
一开始单体直连最简单，但一旦系统开始变复杂（鉴权、限流、降级、灰度、TraceId），把这些逻辑堆在 Master 会导致：
- 业务代码混入大量“入口治理逻辑”，维护成本暴涨
- 安全风险（比如 Master 全放行）更难控制

引入 Gateway 的好处是把职责拆清楚：
- Gateway：处理“入口”相关的共性问题（鉴权/限流/降级/灰度/TraceId）
- Master：只专注业务与调度（任务、状态机、DB、RPC）

代价是：多一个服务、多一层转发、多一套配置和排障链路。

### 2.2 为什么“Master 信任网关”必须要有签名
如果 Gateway 只是简单透传 `X-User-Id`，任何人都能伪造 header 直打 Master。

所以今天的关键升级是：**Master 不相信任何外来 header，只相信“带网关签名且校验通过”的 header**。
这属于“信任边界”的正确建立方式。

### 2.3 为什么我们没有直接做 MDC 传播
MDC 传播能把 traceId 带到异步线程里，但也会带来风险：
- ThreadLocal 泄漏导致串台（最可怕：排障被误导）
- 实现复杂度上升，容易掉坑

所以我们选了更稳的折中：
- HTTP 链路用 traceId
- 任务日志第一行写 `TRACE_ID:` 建立关联
- 后续再按需做 MDC 传播（写进 Future Work）

### 2.4 为什么“Master 不主动连接 Worker”
今天最重要的架构认知：在 Netty 长连接模型里，
**正确的是 Worker 主动连接 Master，Master 在已连接的 Worker 池里选择 idle 下发任务**。

如果要让 Master 去“找 Worker 并连接”，那 Worker 也必须开 server 端口，涉及：
- NAT/防火墙/地址发现/安全控制
- 成本大、复杂度高、很不适合一开始就上

## 3. 实现细节：我学到的关键工程技巧

### 3.1 限流与降级不是“写个 if”，而是系统能力
限流（Redis 原子计数）让系统有“自我保护”：
- 防止 submit 被刷导致 Master/DB/Worker 全崩

降级（超时 + 统一 fallback）让系统有“更好的失败体验”：
- 上游不可用时不让用户白屏/卡死
- HTML 请求跳转到友好错误页，API 返回 503 JSON

### 3.2 TraceId 的真正用法
TraceId 不存数据库，价值在于：
- 用户报错给你一个 `X-Trace-Id`
- 你在日志里按 `[traceId]` 一搜就能定位整条请求链路

今天还做了更实用的一步：
- 任务日志第一行写入 `TRACE_ID:<traceId>`
- 以后可以从任务日志反查入口请求

### 3.3 解决“任务被遗忘”的两条腿：队列 + 对账
今天的关键修复思路：
- **队列**：调度失败时入队（去重），worker 空闲/任务结束触发 pop 下发
- **对账（只入队）**：定期扫描 DB PENDING，在存在 idle worker 时补入队列，防止丢失

这让我理解了“事件驱动 vs 定时对账”的配合：两者缺一不可。

### 3.4 解决“RUNNING 卡死”的思路：租约 + 回收
当 Worker 宕机，任务可能永远 RUNNING。
正确做法是：
- Worker/任务的“占用状态”由 Redis 租约表达（TTL + 续租）
- 租约断供后由 Master 回收（RUNNING -> PENDING 入队）

### 3.5 attemptId（栅栏 token）的核心：允许重复跑，但不允许写错结果
分布式系统里在网络抖动/宕机下很难完全避免重复执行。
attemptId 的目标是“最终一致性保险”：
- Master 每次下发前从 Redis `INCR` 得到 `currentAttempt`
- Worker 上报状态携带 attempt
- Master 只接受 attempt == currentAttempt 的结果

这相当于给“结果写入权”加了一把锁。

## 4. 今天踩过的坑与解决方式（经验库）

### 4.1 网关错误页重定向死循环
现象：浏览器提示“重定向太多次”。
原因：fallback 跳转的错误页又被网关转发/鉴权拦截，导致无限循环。
解决：
- 错误页从转发路由中排除
- 错误页加入鉴权白名单
- fallback 逻辑排除错误页路径

### 4.2 Worker 一直重连但连不上 Master
现象：Worker 不停打印重连日志。
原因：Master RPC 端口 9000 没监听（被配置禁用），或者 Worker 连错 host/port。
解决：
- 明确 `rpc.server.enabled` 开关并检查 Master 日志
- Worker 支持通过 env/args 配置 Master host/port，避免写死 localhost

### 4.3 scheduled 定时任务在测试环境误触发
现象：H2 没建表，定时任务查询报错。
解决：测试 profile 里关闭调度相关定时任务开关。

## 5. 今天接触到的新技术栈（从“会用”到“理解”）

### 5.1 Spring Cloud Gateway（网关）
- GlobalFilter / 路由规则 / 响应头注入
- 限流、降级、灰度这些“平台能力”的落点

### 5.2 Nacos（服务发现）
- Gateway 通过 `lb://serviceName` 转发到 Master
- Canary 通过注册不同 serviceName 实现分流

### 5.3 Netty + Protobuf（RPC）
- 长连接通信
- 用消息类型区分心跳、下发任务、日志流、状态上报
- “连接模型”理解：Worker 主动连，Master 选连接下发

### 5.4 Redis（分布式协调）
- Lua 原子抢占（避免并发抢同一 worker）
- 租约（TTL + 续租）表达“占用/存活”
- 队列（list + set 去重）表达“等待调度”
- attempt（INCR）表达“当前执行版本”

1. 先补齐“异常/宕机/重启”的自愈能力（队列/对账/回收/attempt）
